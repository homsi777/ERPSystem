# WPF Performance Rescue — Phase A: Diagnostic Report

**Date:** 2026-07-11
**Scope:** WPF desktop app only (`ERPSystem.dll`, `net9.0-windows`). React web-client and `ERPSystem.Api` untouched and out of scope.
**Rule followed throughout:** zero changes to financial/accounting/inventory calculations or outcomes. Every change made in this phase was additive instrumentation only (see "Instrumentation added" below) — no query result, no posted amount, no stock quantity, no query filter/order was altered.

---

## 0. Executive summary

The "3 minutes to open Sales" complaint is real and **measured**, but it is not caused by the Sales screen's own queries — those are already reasonably close to target. The dominant cost is paid **once per app launch, before any screen is shown**, and then compounds with a handful of genuine N+1 patterns once the user starts navigating:

| Stage | Measured cost | Root cause |
|---|---|---|
| 1. SSH tunnel to the remote PostgreSQL VPS (only if not already up) | up to ~20s, blocking, on the UI thread | `Thread.Sleep` polling loop in `SshTunnelService` |
| 2. DB migrate + seed + accounting-health validation | **48,064 ms measured**, 225 DB round trips, 38,663 ms of that is pure DB wait | N+1 loop in `DatabaseSeeder.EnsurePermissionsAsync` (~180 of the 225 round trips) |
| 3. `MainWindow` construction (eager build of all 12 top-level modules + their default subpages) | **3,953 ms measured**, only 360 ms of which is DB — the rest is pure UI-thread control construction | All 12 modules constructed eagerly in `MainWindow()`'s constructor instead of lazily on first navigation |
| 4. First Sales invoice list load | **~1,020-1,477 ms measured** (6 DB round trips, ~1,000-1,100 ms of DB wait) | Not an N+1 — already well-batched — but 3 independent lookups are awaited sequentially instead of in parallel, and every round trip pays the full network latency to the VPS |
| 5. Opening one invoice's Operations Center | **5,658 ms measured** (17 DB round trips) | Confirmed N+1 — journal entries re-fetched one-by-one in a loop |

Stages 1-3 alone already account for roughly **50-70 seconds before the user can click anything**, on every single launch. Stack a couple of Operations Center opens (≈5-6s each) and a return to the Sales tab that reloads from scratch (no cross-navigation caching — Finding N-2.2/2.5), and a real user's "it took 3 minutes to get to my invoices" becomes fully explainable **without needing to invoke slow database hardware or huge data volumes** — the actual production `erp_pro` database is tiny (4 sales invoices, most tables near-empty per `_perf_audit.txt`).

*Every finding in this report was produced by this session's own direct code reading and live measurement, then independently cross-checked by two additional automated read-only audit passes (N+1 patterns; DataGrid virtualization + navigation). All three converged on the same core root causes and the automated passes additionally surfaced further instances of the same patterns in other screens — folded into §4 below.*

**The single fact that explains almost everything else in this report:** every database round trip on this connection costs on average **~250ms** (measured: min 128ms, max 876ms, avg 249ms across 69 real queries — see §2). On a local/LAN database this would be closer to 1-2ms. This 100-250x latency multiplier is why *any* sequential/N+1 pattern that would be invisible on a local dev machine becomes a multi-second-to-multi-minute problem here.

## 2. Instrumentation added (Phase A deliverable)

A new, purely observational instrumentation layer was added under `Diagnostics/Performance/`:

| File | Purpose |
|---|---|
| `IWpfPerformanceProfiler.cs` / `WpfPerformanceProfiler.cs` | Central profiler: `BeginScreenLoad(screenName)` returns a `WpfPerformanceScope`; records total time, DB query count/wait time, rows returned, and classifies severity against the task's thresholds (Warning ≥100ms, High ≥500ms, Critical ≥1000ms; screen-load target 1500ms). Logs via `ILogger` and best-effort appends JSONL to `%LocalAppData%\ERPSystem\perf-logs\`. |
| `WpfPerformanceScope.cs` | `IDisposable` scope with `Mark*`/`Measure*` helpers for sub-phases (data load, mapping, items-source assignment, DataGrid binding, rendering). Any phase not marked is simply omitted — tolerant of the codebase's code-behind (View+ViewModel combined) style. |
| `EfQueryTelemetry.cs` | Counts EF Core round-trips **and** cumulative DB wait time per active scope by subscribing to the standard .NET `DiagnosticListener` feed that EF Core already publishes (`Microsoft.EntityFrameworkCore.Database.Command.CommandExecuted`) and correlating via `AsyncLocal<T>`. **No changes to `ErpDbContext` or Infrastructure DI registration were made** — this is 100% additive and cannot alter query behavior. No SQL text or parameter values are captured, only a count and a duration. |
| `ScreenLoadMetric.cs`, `NavigationMetric.cs`, `UiThreadMetric.cs`, `PerfSeverity.cs` | Plain data records for the three kinds of measurement the task asked for (screen loads, navigation, UI-thread operations). Never store customer/financial/inventory data — only screen names, counts, and durations, so they are safe to log. |

**Wired into (all additive, zero behavior change):**
- `App.xaml.cs` — wraps `MigrateAndSeedAsync` + `CurrencyCatalog.RefreshAsync` in an `App.Startup` scope, and wraps `new MainWindow(); .Show();` in an `App.MainWindowConstruction` scope.
- `Controls/Sales/SalesInvoiceListPageControl.cs` — wraps `LoadInvoicesAsync` (the priority screen named in the task) in a `Sales.InvoiceList` scope with data-load/mapping/items-source phases.
- `Controls/Sales/SalesInvoiceOperationsCenterControl.cs` — wraps `ReloadAsync` in a `Sales.OperationsCenter` scope.

This instrumentation is safe to leave in the codebase permanently (near-zero overhead — a handful of `Stopwatch` reads and an `AsyncLocal` counter per screen load) and is the foundation for measuring every future optimization's actual impact, and for extending measurement to the remaining screens in Phase B without re-deriving the pattern.

| Screen / stage | Total time | DB round trips | DB wait time | Rows | Classification |
|---|---|---|---|---|---|

### 4.3 DataGrid virtualization & navigation/duplicate-load audit — `artifacts/wpf-datagrid-and-navigation-audit.md`

This audit was likewise cross-checked by an independent automated pass, which refined the initial "clean" verdict and added several navigation findings.

- **DataGrid virtualization: mixed.** The shared list-page host (`ErpListModuleControl`, used by Sales/Customers/Suppliers/Purchases/Expenses/Capital Partners lists) is confirmed clean — `EnableRowVirtualization="True"`, no `"Auto"` column widths, no extra outer `ScrollViewer`. However, several **custom-built** screens do wrap their `DataGrid` in an outer `ScrollViewer`, which silently disables row virtualization regardless of the property setting: `NewSalesInvoiceControl` (invoice line items — **Critical**), `CustomerAccountStatementControl` (ledger — High), `WarehouseDetailingPageControl` (High), `AgingListControls` (High, compounds with the unbounded page-size finding), `ModuleReportViewControl` (High), `SalesTaxReportPageControl` (Medium).
- **Navigation/duplicate-load:**
  - Same-subpage re-navigation is already de-duplicated (good, `ModuleShellControl.SelectSubpage`).
  - Every navigate-away-and-back rebuilds the subpage from scratch — confirmed to apply to **every** module's view factory, not just Sales (Medium-High).
  - A single-row action (approve/edit/cancel) triggers a full list reload via `SalesListRefreshHub` instead of an in-place patch (Low/Medium).
  - **Eager construction of all 12 top-level modules (+ their default subpages) in `MainWindow()`'s constructor — measured at 3,953ms, ~3.6s of which is pure UI-thread control construction with no data loading involved.** (High — see §3.)
  - **9 list controls subscribe to a `*RefreshHub` event without unsubscribing on `Unloaded`** (Cashbox, CashboxTransfer, OpeningBalance, InventoryWarehouse, 3× inside `InventoryFabricStockPageControl.cs`, InventoryFabricCategories, PurchaseInvoiceList, Employee, Department) — combined with the always-recreate-on-navigation finding above, these leak and accumulate duplicate handlers over a session (High).
  - Dashboard reloads its data from 3 independent, overlapping triggers (`Loaded`, `LanguageChanged`, `ErpDataRefreshHub`) instead of one debounced entry point (High).
  - `NewSalesInvoiceControl` loads its lookups in both `OnLoaded` and `OnIsVisibleChanged` — a direct duplicate load (High).
  - Warehouses/tax codes/branches are re-fetched from the DB on every screen that needs them; only currency is cached today via `CurrencyCatalog` — that pattern should be extended (Medium).

## 6. What Phase B should fix, in priority order

See `WPF_PERFORMANCE_REMEDIATION_PLAN.md` for the detailed, sequenced plan. In one line each, ordered by measured impact:

1. Fix the seeder's permission loop (Critical, ~38s of the ~48s startup cost).
2. Make the SSH tunnel wait async instead of `Thread.Sleep`-blocking the UI thread at startup.
3. Make top-level module construction lazy instead of building all 12 eagerly (High, ~3.6s).
4. Fix the Sales Operations Center journal-entry N+1 (High, 5.7s → estimated <1s).
5. Parallelize the Sales invoice list's 3 independent lookups (Medium, easy, zero risk).
6. Cache subpages across navigation instead of rebuilding on every visit (Medium).
7. Fix the two Excel-parsing sync-over-async handlers (High for large files, no impact on today's small-file usage).
8. Bound the Aging report's page sizes / move aggregation server-side (High, grows worse over time).
